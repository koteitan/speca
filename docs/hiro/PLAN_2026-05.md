## 4. Track 3 — Phase B プロンプト自己改善 (RSI 統合)

### 3.1 設計方針

「プロンプトが不完全でもアウトプットを改善する」を **2層構造** で実現:

| 層 | 動作タイミング | 役割 | 既存研究のベース |
|---|---|---|---|
| **Inner loop** (実行時) | 1 batch 内 / 1 turn 内 | LLM が自身の出力を批評 → 改稿してから返す | Self-Refine, Reflexion, RCI |
| **Outer loop** (学習時) | バッチ間 / 日次 | held-out 検証セットで prompt そのものを進化させる | OPRO, Promptbreeder, DSPy, TextGrad |

### 3.2 RSI / Self-improvement 文献サーベイ (W3-W4 で実施)

#### A. Inner loop 候補

| 手法 | 出典 | 特徴 | SPECA 統合先 |
|---|---|---|---|
| **Self-Refine** | Madaan et al. 2023 (NeurIPS) | LLM が自身の出力に feedback → refine をループ | Phase 03 worker, Phase 04 review |
| **Reflexion** | Shinn et al. 2023 (NeurIPS) | 失敗の verbal reflection を episodic memory に蓄積 | Phase 04 reviewer (FP パターン蓄積) |
| **RCI (Recursive Criticism & Improvement)** | Kim et al. 2023 | LLM が自身の出力を critique → improve | Phase 03 finding emit 前 |
| **Tree of Thoughts** | Yao et al. 2023 (NeurIPS) | 思考枝の探索 + 自己評価 | Phase 03 hypothesis enumeration |
| **AlphaCodium** | Ridnik et al. 2024 | flow engineering for code, iterative test→fix | Phase 05 PoC 生成 |

#### B. Outer loop 候補

| 手法 | 出典 | 特徴 | SPECA 統合先 |
|---|---|---|---|
| **OPRO** | Yang et al. 2023 (Google) | LLM をオプティマイザとして prompt を進化、過去スコア+候補をプロンプトに渡す | prompt iteration の baseline |
| **Promptbreeder** | Fernando et al. 2023 (DeepMind) | 進化アルゴリズム + LLM mutation で prompt 集団を進化 | 多様性確保したい場合 |
| **DSPy** | Khattab et al. 2023 (Stanford) | プログラム的に prompt+demos を最適化、teleprompter | パイプライン全体の coordinated optimization |
| **TextGrad** | Yuksekgonul et al. 2024 (Stanford) | テキスト勾配で逆伝播、改善方向を自然言語で算出 | per-phase prompt の細粒度修正 |
| **SAMMO** | Schnabel & Neville 2024 (MSR) | 構造を保ったまま multi-objective 最適化 | コスト×精度のパレート探索 |
| **ADAS** | Hu et al. 2024 | エージェント設計自体を LLM が探索 | Phase 構成・worker 数の自動調整 |
| **Eureka** | Ma et al. 2024 (NVIDIA) | code-as-reward + 進化、ロボット制御で実証 | reward 設計 (recall vs precision tradeoff) |

#### C. 関連 (理論 / 背景)

| 文献 | 補足 |
|---|---|
| STaR (Zelikman 2022) | rationale 自己生成 → fine-tune (我々は fine-tune しないが思想は参考) |
| Constitutional AI (Bai 2022) | self-critique の原則化 |
| Recursive Self-Improvement (Bostrom / Yudkowsky) | 高水準理論、安全性議論 |

#### D. サーベイ方法 (W3 で実施)
1. Google Scholar で各論文の被引用論文を 2 階層辿る
2. arXiv の "self-improvement" / "prompt optimization" / "LLM iterative refinement" カテゴリを直近 18ヶ月走査
3. 重要 GitHub repo のスター数推移と最終コミット日を記録 (生きてる実装か判定)
4. SPECA への適用可能性を 5 段階評価 (Inner/Outer × 効果想定 × 実装コスト)
5. 結果を `speca-audits-2026/research/rsi_survey.md` に格納

### 3.3 既存ワークフロー / script / CLI への統合

#### Inner loop 統合点

```
scripts/orchestrator/runner.py
  ClaudeRunner._execute_batch
    → 既存の retry layer に "self-critique turn" を挟む
    → max_turns_per_batch を 1 増やす分だけコスト増
    → 設定: PhaseConfig.inner_loop_strategy = "none" | "self_refine" | "reflexion"

prompts/03_auditmap_worker_inline.md
  → "ここまで書いた内容を、別人格として critic role で読み直し、
     反証可能性 / scope 適合性 / 重複可能性 をチェックしてから最終出力せよ"
  → Phase 04 の 3-gate 後段では Reflexion 風メモリを使う

prompts/04_review_worker.md
  → 過去 batch の DISPUTED_FP パターンを episodic memory として注入
  → memory store: outputs/.reflexion_memory/<phase>.jsonl
```

#### Outer loop 統合点

```
benchmarks/prompt_iteration/   # 新規ディレクトリ
  ├── eval_harness.py          # introduced_in_commit time-slice replay
  ├── opro_optimizer.py        # OPRO 実装 (Phase 別)
  ├── promptbreeder.py         # 進化的 prompt 探索 (任意)
  ├── dspy_pipeline.py         # DSPy 経由の coordinated 最適化
  └── runs/                    # 各イテレーションの結果 (precision/recall/cost)

scripts/run_prompt_iter.py     # 新規 entry point
  --phase 03|04
  --strategy opro|promptbreeder|dspy
  --rounds 3
  --eval-slice held_out_2025_q4
  --budget-usd 300

speca-cli prompt-iter          # 新規 subcommand (M3 以降)
  CLI 内で iteration 起動 + 改善カーブ可視化 + best prompt promote
```

#### 評価ハーネス (Outer loop 必須)
- 入力: `csv/past_defi_patterns.csv` (時系列分割) or `speca-audits-2026/dataset/` の time-sliced split
- メトリクス: recall (vendor 公表脆弱性に対する命中率) / precision (DISPUTED_FP 率) / cost ($/run)
- ベースライン: 現行 prompts (untouched)
- 検証スライス: held-out 100件 (各カテゴリ stratified)
- ロック条件: held-out で baseline + ≥10pp recall、precision 退化なし、cost 1.5x 以内
- ロック先: `speca-audits-2026/prompts/eth-audit-2026/locked/<phase>_v<n>.md`

#### CLI 連携 (M3 以降)
- `speca prompt-iter` サブコマンド (上記)
- dashboard 上に "Iteration progress" ペイン (round 数 × 改善 delta)
- 改善カーブをローカル sparkline で表示
- locked tag を CLI から `speca prompts apply <tag>` で全 phase に反映
