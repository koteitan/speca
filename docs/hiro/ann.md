## 1. 失敗の定量化と分類パイプライン

> 「失敗を定量化する → 失敗したのを振り分ける → 成功したのを振り分ける」

### 概要
検出結果を成功/失敗に自動分類し、失敗パターンを定量化するフィードバックループ。

### 具体案
- Phase 04 (Review) の DISPUTED_FP / CONFIRMED をさらに **失敗原因別に分類**
  - 検出漏れ (FN): なぜ見逃したか → プロンプト不足? コンテキスト不足? モデル限界?
  - 誤検出 (FP): なぜ誤報したか → 幻覚? スコープ外? Dead code?
- 失敗メトリクスのダッシュボード化
- 失敗パターンの **自動フィードバック** → 次回実行のプロンプト改善

### ベンチマーク観点
- 失敗分類の分布を CWE 別・Phase 別に集計
- 「どの種類の脆弱性で失敗しやすいか」の傾向マップ

## 2. 自己修復フレームワーク (Self-Healing)

> 「自動で目的に適してフレームワークに変化させる」

### 概要
パイプラインが失敗・停滞した際に、自動的に戦略を変更して再試行するメタ制御層。

### 具体案
- **プロセス停止時の自動リカバリ**: 最後の 50 行を次のエージェントに渡して継続
  - 「最後に分かったことをまとめさせて、構造化して次に引き継ぐ」
  - 現在の resume.py の拡張: 部分結果だけでなく **失敗コンテキスト** も引き継ぎ
- **プロンプト自動調整**: 失敗パターンに基づいてプロンプトを動的に書き換える
  - 例: CWE-362 (Race Condition) の検出率が低い → 並行性分析の指示を強化
  - A/B テスト的にプロンプトバリアントを試行
- **Phase 構成の動的変更**: 特定ターゲットに対して Phase の順序や深さを変える

### 実装イメージ
```
[Orchestrator] → 実行 → 失敗検出
                          ↓
                  [Self-Heal Controller]
                          ↓
                  分析: 「CWE-362 で FN 多発」
                          ↓
                  対策: プロンプトに並行性チェック追加
                          ↓
                  [Orchestrator] → 再実行
```

## 3. プロンプト可変フレームワーク

> 「プロンプトを変えるフレームワーク」

### 概要
状況に応じてプロンプトテンプレートを自動選択・合成するシステム。

### 具体案
- **プロンプトライブラリ**: CWE 種別ごと、言語ごと、ドメインごとの専門プロンプト群
- **動的合成**: ターゲットの特性を分析して最適なプロンプトを組み立てる
  - 入力: ターゲットの言語、フレームワーク、CWE 分布
  - 出力: カスタマイズされた Phase 03 プロンプト
- **進化的プロンプト最適化**: 成功率が高いプロンプトの要素を学習して次世代に反映

### ベンチマーク観点
- プロンプトバリアント A vs B の検出率差を統計検定 (McNemar)
- 最適プロンプトの発見過程を定量化

## 4. ソーシャルエンジニアリング手法の応用

> 「ソーシャルエンジニアリング手法」
> 「あえて攻撃と防御に分かれるんだけどその時に内通者を作って効率よく分析」

### 概要
攻撃者の心理・手法をエージェントに模倣させることで、従来の静的解析では見つからない脆弱性を発見。

### 具体案
- **攻撃者ペルソナ**: エージェントに「あなたは熟練の攻撃者です」とロール設定
  - 通常のコードレビューとは異なる視点で脆弱性を探索
  - 「どうやってこのシステムを破るか」から逆算
- **内通者シミュレーション**: 防御側エージェントの中に「意図的に弱点を探す」エージェントを配置
  - 防御が見落としている箇所を効率的に発見
- **Red Team / Blue Team 自動化**: 攻撃エージェント vs 防御エージェントの対戦ログから学習

## 5. マルチ AI 比較・使い分けシステム

> 「AI 同士を比較して、どれがどこだけ優れてて、どんな AI はどこに適しているのかみたいなのを使い分けるシステム」
> 「そのデータセットとかで殺すシステムとどんくらい殺せたのかみたいなのを出力する」

### 概要
複数の AI モデル/エージェントを同一タスクで競わせ、得意分野を特定し、最適なルーティングを行う。

### 具体案
- **モデル別ベンチマーク**: 同一 PrimeVul データセットで Claude / Codex / GPT / Gemini を実行
  - CWE 別の検出率比較 → 「CWE-787 は Claude が強い」「CWE-362 は Codex が強い」
  - コスト対効果の比較 ($/detection)
- **動的ルーティング**: 脆弱性の種類に応じて最適なモデルに振り分け
  ```
  入力: コードスニペット + CWE ヒント
    → Router: 「メモリ安全性 → Claude Sonnet」
    → Router: 「並行性 → GPT-4o」
    → Router: 「ロジック → Claude Opus」
  ```
- **アンサンブル投票**: 複数モデルの結果を集約 (majority vote / weighted vote)
- **出力**: モデル別 kill rate (検出率) のリーダーボード

### ベンチマーク観点
- PrimeVul / CVEFixes でモデル間比較テーブル
- CWE×モデル のヒートマップ
- コスト効率 (detections per dollar) の比較

### Codex / 他モデルとの連携
> 「Codex に振り分ける」「他モデルとの連携」

- **OpenAI Codex** をコード理解・パッチ生成に特化して使用
- OpenAI o-series / GPT-4o をロジック分析・並行性解析に活用
- SPECA の Phase 05 (PoC 生成) を Codex / 他モデルに委任
- Claude = 監査推論、Codex = コード生成・パッチ提案の役割分担

## 6. プロセスリカバリとコンテキスト引き継ぎ

> 「プロセス止まったら最後の 50 行を次の agent に渡して go、最後に分かったこととかまとめさせてね、構造化でもいいけど」

### 概要
エージェントのコンテキストが枯渇 or クラッシュした場合に、知見を構造化して次のエージェントに引き継ぐ。

### 具体案
- **自動サマリ生成**: コンテキスト制限到達時に「これまでの発見」を構造化 JSON で出力
  ```json
  {
    "findings_so_far": [...],
    "current_hypothesis": "...",
    "unexplored_areas": [...],
    "failed_approaches": [...]
  }
  ```
- **チェーンエージェント**: Agent A → サマリ → Agent B → サマリ → Agent C
  - 各エージェントは前任の知見を踏まえて探索を継続
- **現状の resume.py を拡張**: PARTIAL 結果 + 失敗コンテキスト + 仮説の引き継ぎ

## 7. 過去脆弱性 RAG (Retrieval-Augmented Generation)

> 「過去の脆弱性から RAG 作って、似たようなところにまた同じような脆弱性ないか掘らせれば」

### 概要
既知の脆弱性データベースから類似パターンを検索し、ターゲットコードの同種の脆弱性を効率的に発見。

### 具体案
- **脆弱性ベクトル DB**: CVEFixes / NVD / PrimeVul のコードを embedding 化して格納
  - 入力: ターゲットの関数コード
  - 検索: 類似する過去の脆弱コードを取得
  - 出力: 「この関数は CVE-2024-XXXX と構造が似ている。同種の脆弱性がないか確認」
- **パターン学習**: CWE 別の「よくあるコードパターン」を蓄積
  - CWE-787: `memcpy(dst, src, user_controlled_len)` パターン
  - CWE-416: `free(ptr); ... use(ptr)` パターン
- **プロンプト強化**: RAG で取得した類似脆弱性を Phase 03 のプロンプトに注入
  ```
  「以下の過去の脆弱性と類似するパターンがこのコードにないか確認してください:
   [RAG 結果: CVE-2023-1234, CWE-787, strcpy without bounds check]」
  ```

### ベンチマーク観点
- RAG あり vs RAG なしの検出率差
- RAG の precision@k (上位 k 件の類似コードがどれだけ関連するか)

## 8. AI 攻防コンペティション

> 「セキュリティの AI のコンペで防御する AI と攻撃する AI に分かれて戦うみたいなやつで、AI のログ取って学習させる」
> 「それで傾向を殺す」

### 概要
攻撃エージェントと防御エージェントを対戦させ、そのログから両者を強化する GAN 的アプローチ。

### 具体案
- **Red Agent (攻撃)**: 脆弱性を埋め込んだコードを生成 or 既存コードの攻撃ベクトルを発見
- **Blue Agent (防御)**: コードを監査して脆弱性を検出・修正
- **対戦ログ学習**:
  - Red が見つけた攻撃パスで Blue が見逃したもの → Blue のプロンプト改善
  - Blue が検出したが Red が想定していなかったもの → Red の攻撃戦略改善
- **傾向分析**: 「どのモデルがどの攻撃パターンに弱いか」のヒートマップ

### 実装イメージ
```
Round 1:
  Red Agent → 脆弱コード生成 (CWE-89: SQL Injection)
  Blue Agent → 検出試行 → 成功/失敗

Round 2:
  Red Agent → Blue が見逃した種類の脆弱性を生成
  Blue Agent → 前ラウンドのフィードバックで強化

→ N ラウンド後: 両者のログから傾向分析
```

### ベンチマーク観点
- ラウンドごとの検出率推移グラフ
- 攻撃成功率 vs 防御成功率の収束カーブ
- モデル別の攻撃/防御適性スコア

## 9. リバースエンジニアリング → 仕様抽出 → 理想構造比較

> 「リバースエンジニアリングして仕様書抽出して最強の構造と比較して違う時の問題点出す」

### 概要
ターゲットコードから仕様を逆抽出し、理想的なセキュアアーキテクチャと比較して差分を脆弱性候補とする。

### 具体案
- **コード → 仕様の逆変換**: SPECA の Phase 01b (Subgraph Extraction) の逆方向
  - コードから状態遷移図・データフロー図を自動生成
  - 関数間の信頼境界を推定
- **理想構造テンプレート**: CWE Top 25 に対する「安全な実装パターン」のカタログ
  - 認証: OAuth 2.0 のベストプラクティス構造
  - 入力検証: ホワイトリスト方式の標準構造
  - メモリ管理: RAII パターン
- **差分分析**: 逆抽出した仕様 vs 理想構造のギャップ = 脆弱性候補
  ```
  実際: [ユーザ入力] → [直接DB] (検証なし)
  理想: [ユーザ入力] → [サニタイズ] → [パラメータ化クエリ] → [DB]
  差分: 入力検証の欠如 → CWE-89 候補
  ```

## 10. Web アプリ化 / 可視化出力

> 「Web app にするとか、WEB とか AI が認識できる形で出力して分析する」
> 「うまくいかなかったら WEB とかじっさいに見えるところから探すところに始める」

### 概要
監査結果をインタラクティブな Web ダッシュボードで可視化し、人間と AI の両方が分析しやすい形で出力。

### 具体案
- **結果ダッシュボード**:
  - CWE 別ヒートマップ
  - Phase 別パイプラインフロー図
  - Finding ごとのドリルダウン (コード → プロパティ → 根拠)
- **AI 可読出力**: 構造化 JSON / SARIF 形式で出力
  - 他のツール (GitHub Code Scanning, SonarQube) との連携
- **インタラクティブ探索**: 人間が「この関数をもっと深く調べて」と指示できる UI

## 11. Kali Linux ツールオーケストレーション

> 「既存の Kali Linux の〇〇のツールを操作！みたいな、オーケストレーションで」

### 概要
既存のセキュリティツール群を AI エージェントが統合的に操作・解釈するオーケストレーション層。

### 具体案
- **ツール統合**: Nmap, Burp Suite, Metasploit, sqlmap 等を MCP サーバ経由で操作
- **AI 判断ループ**:
  ```
  Agent: Nmap でポートスキャン → 開いてるポートを発見
  Agent: Burp Suite で HTTP エンドポイントをクロール
  Agent: sqlmap で SQL Injection テスト
  Agent: 結果を統合して脆弱性レポート生成
  ```
- **SPECA との統合**: Phase 01a (Spec Discovery) の入力として動的スキャン結果を利用

## 12. マルチペルソナレビュー

> 「同じものをいろんな役職の人に振り分けて、出させて、待機して全部で案が出るみたいな」

### 概要
同一コードを複数の専門家ペルソナに同時レビューさせ、結果を集約。

### 具体案
- **ペルソナ例**:
  - セキュリティアーキテクト: 設計レベルの脆弱性
  - ペネトレーションテスター: 攻撃者視点の実行可能性
  - コードレビュアー: 実装レベルのバグ
  - コンプライアンス監査人: 規制要件との適合性
  - DevOps エンジニア: 設定・デプロイの脆弱性
- **集約方法**:
  - 全ペルソナの結果を待機 (parallel execution)
  - 重複検出を統合、ユニーク検出を追加
  - 信頼度のアンサンブル (3/5 のペルソナが指摘 → 高信頼度)

### ベンチマーク観点
- 単一ペルソナ vs マルチペルソナの検出率差
- ペルソナ数と検出率の関係 (diminishing returns の閾値)
- ペルソナ間の一致率 (Cohen's kappa)

## 13. リアルタイムベンチマーキング

> 「結果を分析みたいな」「コンペ中とか作業中にベンチマークを取る」

### 概要
監査実行中にリアルタイムでメトリクスを計測・表示し、動的に戦略を調整。

### 具体案
- **ライブメトリクス**: Phase 実行中に検出率・FP 率・コストをストリーミング表示
- **早期停止**: FP 率が閾値を超えたら自動停止 → プロンプト調整 → 再開
- **A/B テスト実行**: 同一バッチの半分を戦略 A、半分を戦略 B で実行して比較
- **現状の watchdog.py を拡張**: CostTracker に加えて DetectionTracker を追加

## 14. 効率的なデータ収集

> 「効率よくデータを取る」

### 概要
ベンチマーク用の脆弱性データセットを効率的に収集・整備するパイプライン。

### 具体案
- **自動ラベリング**: CVE → コミット → 関数レベルの before/after ペア抽出の自動化
- **アクティブラーニング**: 検出が不確実なサンプルを優先的に人間レビューに回す
- **データ品質メトリクス**: ラベルの一貫性チェック、重複検出、分布バランスの監視
- **CVEFixes / PrimeVul / Vul4J の統合**: 複数データセット横断の統一フォーマット
