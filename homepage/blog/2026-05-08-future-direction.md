{/* truncate */}

## 1. 複数 AI エージェント環境への対応

SPECA のコア思想は AI エージェントが仕様から脆弱性を検出する仕組みですが、現在は Claude Code を前提に設計されています。今後、Codex CLI や Gemini CLI といった他のエージェント環境でも同じフレームワークを実行できるようにしたいと考えています。

ハーネス設計を通じて、異なるモデル・プロバイダに対応可能な抽象化層を作ることで、Anthropic の認証インフラへの依存リスクも低減できます。複数のエージェントプラットフォームをサポートすることで、セキュリティ業界全体への波及効果も期待しています。

## 2. コミュニティ駆動型マーケットプレイス

仕様ファイル → Property Mapping のナレッジを、コミュニティ内で共有・再利用できる環境を構想しています。複数の監査人が同じプロトコルのマッピングを作成すれば、品質の高い定義が自動的にベンチマークで選別される仕組みです。

こうすることで、SPECA の audit ノウハウが個人に閉じるのではなく、参加者全員の資産になります。他者が作った優秀なプロパティ定義を活用すれば、新しい監査対象に対する Time To Audit も大幅に短縮できるはずです。

## 3. 2 つの推進ルート

SPECA の成長には 2 つの異なるアプローチが必要だと考えています。ひとつはコミュニティ主導でしかできない普及方向で、検査ノウハウの共有やボトムアップでの拡張です。もうひとつは、セキュリティ業界の大手プレイヤーとの連携を通じた発展方向で、大規模インフラ、豊富なバグ例データセット、そして継続的な R&D 投資が必要なアプローチです。

この 2 つを並行して進めることで、小規模チームでは実現不可能な領域も視野に入れられると考えています。

## 4. 再帰的自己改善ループ

既存の Phase B prompt iteration loop をさらに発展させ、過去の audit 結果を自動的に分析して SPECA 自身のプロンプトとフィルタ定義を改善する仕組みを組み込みたいと思っています。検出精度が高かった property mapping や false positive が多かった判定パターンを学習することで、モデル性能に依存しない継続的な精度向上が期待できます。

## 5. 自動クロール監査エージェント

GitHub などの公開リポジトリを定期的に巡回し、候補バグを自動検出・報告するエージェント群を構想しています。bug bounty 活動を半自動化することで、セキュリティ研究者の手作業負荷を減らしつつ、より多くのプロジェクトをカバーできる体制を目指しています。

## 6. GUI だけで操作できるインターフェース

現状の SPECA は CLI / TUI 中心の設計で、Claude Code や uv のコマンド操作に慣れている人を前提にしています。一方で、セキュリティ監査の現場には CLI に詳しくない担当者やレビュアもいるので、ターミナルを開かずブラウザだけで監査の起動・進捗確認・結果ブラウズができる GUI 版も用意したいと考えています。

speca-cli の TUI を Web 化する形でも良いですし、別実装でローカル Web サーバ + ブラウザ UI を提供する形でも良いと思います。CLI でしか触れない状態だとコミュニティが広がりづらいので、入口を複数用意するのは早めに着手したいテーマです。

---

これらの構想は段階的に実装していく予定です。SPECA がコミュニティと企業の両面で成長し、AI エージェントによるセキュリティ監査の標準的な手法として定着することを目指しています。
