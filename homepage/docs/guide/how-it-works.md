# どうやって動く？

SPECA は 6 つの段階を順に実行します。それぞれを日本語で説明してから、内部での技術的な名前を紹介します。

## 1. 仕様書を集める

インターネットからプロジェクトの仕様書やドキュメント（GitHub Issues、スコープ定義など）を自動で探し出します。（内部では **Phase 01a** と呼ばれます）

## 2. 仕様書を読み解く

集めた仕様書を図に変換します。「ユーザーが login を呼ぶ」「その後 process_data が実行される」といった状態遷移や処理の流れを、コンピュータが理解しやすい形に整理します。（**Phase 01b**）

## 3. セキュリティの観点でチェック項目を作る

「認証の前に機密データにアクセスされないか」「権限チェックが漏れていないか」といったセキュリティ上の重要な条件（これを「セキュリティプロパティ」と呼びます）を、仕様から自動で導き出します。STRIDE（脅威モデリングの一種）や CWE Top 25（よくある脆弱性のリスト）を参考に判定します。（**Phase 01e**）

## 4. コードのどこを見るか決める

セキュリティプロパティを確認するために、実装されているコードのどこに注目すべきかを事前に探しておきます。「認証処理は auth.py の authenticate 関数」「データアクセスは database.py の fetch 関数」という具合に、コード上の場所を地図のように整理します。（**Phase 02c**）

## 5. 実際にコードを調べる

いよいよコードを読みます。セキュリティプロパティが実際に成立しているか、論理的に証明してみます。「本当に認証後にのみアクセス可能か」を詳しく調査し、証明できない部分（穴）があればそこが候補検出となります。（**Phase 03**）

## 6. 偽の検出を間引く

Phase 03 で見つかった候補には、誤検出（実は問題ではないもの）が混じっていることがあります。3 つのゲート（死んだコード判定、信頼境界チェック、スコープチェック）で検証し、本当の脆弱性だけを残します。（**Phase 04**）
